## 1. How locale flips at runtime

`src/i18n/index.ts` sets up i18next, picks a starting language (URL `?lang=` → `localStorage` → browser → English), and applies `<html lang dir>` on every change. The CSS in `src/styles/rtl.css` keys off `[dir="rtl"]` so layout mirroring is fully declarative — no React-side direction toggles required.

The language switcher (`src/components/LanguageSwitcher.tsx`) is rendered in the sidebar (`src/more-screens.jsx`). Two-button toggle (`EN` / `עב`) calls `changeLanguage` which fires the i18next event that persists the choice and re-renders every `useTranslation()` consumer.

## 2. Adding a new locale

1. Add a JSON file under `src/i18n/<lang>.json` mirroring `en.json`'s key tree.
2. Add the language code to `SUPPORTED_LANGUAGES` in `src/i18n/index.ts`.
3. Add the resource: `void i18n.use(initReactI18next).init({ resources: { ..., <lang>: { translation: <import> } } })`.
4. Add a label entry to `LABELS` in `LanguageSwitcher.tsx`.
5. If the locale is RTL (Arabic, Persian, Urdu), update `applyDocumentDirection` so `dir="rtl"` is set for it as well as Hebrew.

## 3. Extracting strings from a `.jsx` screen

The kiosk's existing screens have inline literal strings throughout (~hundreds across `screens.jsx`, `console.jsx`, `console-ext.jsx`, `more-screens.jsx`). The mechanical workflow:

1. `import { useTranslation } from "react-i18next"` at the top of the screen.
2. Inside the component: `const { t } = useTranslation();`.
3. For each user-visible literal: replace `"Submit"` with `{t("section.submit")}`. Pick keys grouped by feature (`enroll.*`, `verify.*`, `errors.*`).
4. Add the English entry to `src/i18n/en.json` and translate to Hebrew in `src/i18n/he.json`.
5. Confirm the screen flips correctly by toggling languages at runtime.

The chrome strings (sidebar nav, common buttons, key error messages) are already extracted under `nav.*`, `common.*`, `errors.*`. Per-screen extraction (`enroll.*`, `verify.*`, `console.*`, `profiles.*`, `deepfakeLab.*`) has key trees defined in `en.json` + `he.json`; wire them up in the relevant `.jsx` files when those screens are touched. Greenfield work should never introduce new literal strings without an i18n key.

### Hebrew translation provenance

The Hebrew strings in `he.json` are functional placeholders. Production deployment to the Israel National Cyber Directorate must have a native-Hebrew-speaker reviewer pass over the file before sign-off — the source plan flagged this as an explicit risk (`Plan.md` §13 risk register). Concretely the workflow is:

1. Engage a localisation contractor or in-team native speaker.
2. Hand them `en.json` + `he.json` side by side.
3. They edit `he.json` directly (it's a flat JSON file) and return.
4. CI rebuilds with the new translations; no code change required.

## 4. Mobile + RTL acceptance criteria

Per `Plan.md` §13.F9.1 + §13.F9.5, the QA matrix to clear at the milestone-close gate:

- Pixel-perfect screenshots at 1920×1080, 1280×800, 1024×768, 768×1024, 414×896, 375×667 — committed under `docs/screenshots/`.
- Hebrew RTL: every visible string in Hebrew; layout mirrored cleanly; native-speaker reviewer signs off.
- Real iPhone (Safari) + real Android (Chrome) smoke test — the 12-step QA protocol passes on both.
- Touch-only navigation works for every interactive element (no hover-only affordances on phone).

The `useMobileViewportClass` hook in `src/app.jsx` and the `body.biovoice-mobile` overrides in `src/styles/responsive.css` collapse the kiosk's three-column console + the DeepfakeLab two-column grid into single-column scrollable panels below 768 px. Sidebar collapses into a top-row scrollable panel.
