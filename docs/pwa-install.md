# Installing BioVoice as a Progressive Web App

The kiosk ships a Web App Manifest + Service Worker (v1.1.0), so any modern browser can install it as a standalone "app" — fullscreen, dock/home-screen icon, no browser chrome. No app-store account or review.

> **Requirement**: the kiosk must be served over HTTPS (or `localhost`). HTTP-only LAN deployments won't surface the install prompt — the browser disables PWA install on insecure origins by design.

---

## iPhone / iPad (Safari)

1. Open the kiosk URL in **Safari** (Chrome on iOS won't work — Apple restricts PWA install to Safari).
2. Tap the **Share** button (square + arrow at the bottom of the screen).
3. Scroll down → **Add to Home Screen**.
4. Confirm the name "BioVoice" → **Add**.
5. Launch from the new icon. The app opens fullscreen with a black status bar.

Microphone permission prompts work identically to the in-Safari experience. The app uses the same MediaRecorder fallback to `audio/mp4` that already works on iOS Safari.

## Android (Chrome / Edge / Brave)

1. Open the kiosk URL in **Chrome** (Edge and Brave behave the same).
2. Tap the **menu** (⋮ top-right).
3. **Install app** → confirm.
4. The kiosk now appears in your launcher with the BioVoice icon.

Some Android Chrome builds also surface an "Install" prompt automatically when the manifest + service worker pass installability checks.

## Desktop Chrome / Edge

1. Open the kiosk URL.
2. Look for the **install icon** in the URL bar (computer + down-arrow).
3. Click → **Install**.
4. The kiosk opens in its own window, gets a dock/start-menu entry, and behaves like a native app.

## Verifying the install

Once installed, the kiosk should:

- Open in standalone window (no browser address bar).
- Show the BioVoice icon on the home screen / launcher / dock.
- Splash screen on first launch (auto-generated by the OS from the manifest's `theme_color` + 512×512 icon).

If any of those don't happen, run a Lighthouse PWA audit against the deployed URL:

```bash
npx lighthouse@11 https://your-kiosk.example.com --only-categories=pwa
```

A v1.1.0 deploy should score 0.88. The single failing audit is `content-width` — the kiosk is intentionally a 1920×1080 fixed stage scaled to fit, not a responsive layout. Every other PWA criterion (`installable-manifest`, `splash-screen`, `themed-omnibox`, `viewport`, `maskable-icon`) passes.

## What the PWA caches (and what it doesn't)

The service worker (`/sw.js`) precaches the static SPA bundle (HTML + JS + CSS + icons) so the shell loads instantly on repeat visits, even briefly offline.

API endpoints **always pass through to the network**:

| Path prefix | Cached? | Why |
|---|---|---|
| `/health` | No | Liveness probe — must be live |
| `/readyz` | No | Readiness — must be live |
| `/metrics`, `/metrics/summary` | No | Live counters |
| `/users/*`, `/enroll` | No | Profile state |
| `/verify`, `/identify`, `/embed` | No | ML inference |
| `/spoof/*` | No | Spoof generation + scoring |
| `/results` | No | Verification log |

So an installed PWA without network connectivity will boot the UI and surface "OFFLINE" / "WAITING" states for any panel that needs the backend — not silently serve stale data.

## Updating an installed PWA

When you redeploy the kiosk (e.g. v1.1.0 → v1.1.1), the service worker auto-updates on the user's next visit:

1. Browser fetches the new `/sw.js`.
2. New SW takes over (we set `clientsClaim()` + `skipWaiting()` in `frontend/src/sw/sw.ts`).
3. Old precache is cleaned up.

Users may need to close + reopen the PWA window to see the new bundle. There's no app-store review delay.
